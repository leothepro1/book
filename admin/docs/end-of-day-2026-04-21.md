### High-level accomplishments

1. **Migrated production DB from Render Postgres to Neon Postgres**
   (eu-central-1 Frankfurt). Clean cutover, rollback-drill passed both
   directions, Render kept intact as fallback.

2. **Squashed 42-migration history into a single baseline** (110 tables,
   55 enums, 313 indexes, 135 FKs, 1 partial unique index). Fixed 4
   historical drift issues (orphan tables, dead columns, redundant
   indexes, index naming) in the process. Byte-identical semantic
   parity between Render and Neon after squash.

3. **Shipped Core DB Infra Area 1 (Backup)** — nightly pg_dump to
   Cloudflare R2 + manual restore-drill workflow. Verified end-to-end:
   backup creates valid dump, drill restores it into throwaway Neon
   branch, sanity queries pass, branch cleaned up. 6 fix-commits during
   initial validation (each a distinct bug class — no regressions).

4. **Shipped Core DB Infra Area 3 (Health endpoint)** — liveness (edge)
   + readiness (nodejs) + extensible check registry. Found and fixed
   an architectural gotcha as bonus: Vercel functions were running in
   `iad1` (US East) while Neon is `fra1` (Frankfurt). Pinned to `fra1`
   via `vercel.json`. Expected impact: 100ms+ faster p50 on every API
   route, not just health. `/api/health/ready` now reports overall
   `status: ok` with all dependencies under thresholds.

5. **Shipped Core DB Infra Area 4 (Schema drift detection)** — PR-gate
   workflow (migrations ↔ schema.prisma via shadow postgres) + nightly
   cron (schema.prisma ↔ live Neon). Both verified — PR gate correctly
   passes on cosmetic schema changes and fails on unmapped models with
   a readable, actionable diff.

## Known open questions / decisions postponed

### Single-tenant restore mechanism

Scenario C in `db-restore.md` is documented but manual and dangerous.
Shopify's answer is pods-per-tenant architecture; we have
multi-tenant-single-DB. Not a problem at 2 tenants, will be at 100.
No action today — flag for architecture discussion before scaling.

### Vercel plan tier

Current plan capabilities (concurrent functions, memory, cron
frequency) haven't been audited against 10k-tenant scale. Likely fine
at pilot scale, but we haven't verified.

### Neon plan (Scale)

Confirmed we're on Neon Scale ($69/mo min) per db-restore.md
decision tree. Haven't confirmed we actually upgraded from Launch —
the runbook says 30-day PITR assuming Scale. **Double-check on the
Neon dashboard first thing next session.** If still on Launch, the
restore runbook's decision tree overestimates our retention.

### Shadow DB version mismatch (schema-drift-pr workflow)

GHA service uses `postgres:18-alpine`, Neon is `17.8`. The drift
workflow specs PG 18 client against a PG 18 shadow. For pure DDL
parse-and-apply this is fine, but if we ever write a migration that
relies on PG-17-specific behavior, the shadow might accept it when
prod wouldn't. Low risk, noted.

### Sentry context on health endpoint

The readiness route calls `Sentry.captureMessage` on down checks, but
there's no tenant context (health endpoint isn't tenant-scoped).
Might want to add a `health_endpoint: true` tag so these don't
pollute tenant-specific filters in Sentry. Minor.

the whole session but intentionally kept out of scope. It's a local
Claude Code preferences file, not tracked. Don't worry about it.
